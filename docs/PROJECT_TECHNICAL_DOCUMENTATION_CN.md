# RAG2 项目技术文档：运行流程与技术实现

## 1. 文档范围

本文档面向项目维护者、答辩说明、二次开发人员和系统部署人员，描述 `current_rag_pipeline` 项目的整体架构、运行流程、核心模块职责、关键技术实现、质量控制机制和输出产物。

本文档只解释技术设计和运行机制，不包含源代码片段。

## 2. 项目定位

本项目是一个面向行业研究与深度报告生成的本地增强型 RAG 系统。它并不是单一的“向量检索问答”程序，而是一条完整的研究报告生产流水线，主要能力包括：

- 对本地清洗文本资料进行切片、清洗、去噪、结构化和向量化入库。
- 基于 Qdrant 构建本地知识库，支持密集向量、稀疏词法向量、层级父子块召回和重排序。
- 通过 LangGraph 组织多个智能体协作，将复杂问题拆解为研究计划、章节蓝图、证据任务和写作任务。
- 同时调用本地 RAG 与联网 IQS 搜索，形成“本地资料 + 实时网络资料”的混合证据池。
- 将搜索结果标准化为证据包，执行覆盖率评估、补充检索、冲突识别、指标归一和证据分级。
- 基于结构化证据生成行业研究报告，并经过 QA、重写、重排版和引用校验后输出可发布 Markdown。

系统的核心目标是：让最终报告尽量以可追溯证据为依据，减少幻觉、空泛判断、内部调试信息泄露和无来源结论。

## 3. 总体架构

项目可以分为七层：

| 层级 | 主要职责 | 代表模块 |
|---|---|---|
| 启动与编排层 | 加载环境、识别运行模式、调用完整报告或服务入口 | `start_rag.ps1`、`run_full_report.py` |
| 数据入库层 | 文档清洗、语义切片、父子块构建、向量化、同步 Qdrant | `ingest_pipeline.py`、`slicing.py`、`embedding_qdrant.py` |
| 检索层 | 本地混合召回、层级召回、重排序、证据选择、答案综合 | `search/engine.py` |
| 联网搜索层 | IQS 搜索、网页分析、来源过滤、多路联网证据收集 | `web_analysis_agent.py`、IQS lane 节点 |
| 脑智能体层 | 问题分析、研究规划、动态路由、多子智能体并发、证据合并 | `brain_agent.py` |
| 写作生产层 | 证据绑定、证据综合、章节论证、表格、决策、风险、最终写作 | `writer_agent_clean.py` 及 `agents/` 根目录报告生产模块 |
| 质量治理层 | 覆盖率检查、QA、重写、重排版、引用规范化、最终清理 | `qa_agent.py`、`reformatter_agent.py`、`review_agent.py` |

整体运行逻辑可以概括为：

用户问题进入系统后，脑智能体先判断问题类型并生成研究计划。随后系统根据计划拆分出本地检索任务和联网搜索任务。本地 RAG 从 Qdrant 知识库召回证据，联网 IQS 从外部来源补充最新资料。所有子结果统一进入证据池，经过覆盖率评估和补充检索后形成结构化证据包。写作流水线再把证据绑定到章节、构造论证、生成表格和结论，最后通过 QA 与重排版模块输出干净的报告文件。

## 5. 运行入口

### 5.1 完整报告入口

`run_full_report.py` 是生成深度研究报告的主入口。它负责：

- 加载项目根目录下的 `.env` 配置。
- 接收用户问题、股票代码、行业对象、运行模式、输出目录、联网搜索预算等参数。
- 调用脑智能体 `run_brain_agent` 执行完整研究流程。
- 将脑智能体返回的运行状态、写作包和初始报告写入输出目录。
- 在默认情况下继续调用证据提取器和 Reformatter，生成更干净的最终报告。
- 如果 Reformatter 失败，则使用 ReviewAgent 作为兜底清理方案。

完整报告入口默认使用严格质量模式。严格模式会提高联网检索预算、搜索任务数量、候选结果数量、重排序数量和补充检索轮数，目标是生成资料更充分、来源更密集的长报告。

### 5.2 统一启动脚本

`start_rag.ps1` 是本地运行时最常用的统一入口。它的主要逻辑是：

- 定位项目目录和虚拟环境中的 Python 解释器。
- 加载 `.env` 环境变量。
- 自动探测本地 Qdrant 服务地址。
- 设置默认 Qdrant collection 名称。
- 根据首个参数决定运行模式。

它支持的典型模式包括：

| 模式 | 作用 |
|---|---|
| `brain` 或 `agent` | 直接运行脑智能体 |
| `api` | 启动联网搜索相关服务 |
| `serve` | 启动本地 RAG 服务接口 |
| `ingest` | 执行切片和向量入库流水线 |
| `sync` | 将已有切片同步到 Qdrant |
| `benchmark` | 执行检索或报告相关基准测试 |

### 5.3 入库入口

`start_ingest.ps1` 用于构建或刷新知识库。它会：

- 加载 `.env`。
- 检查虚拟环境是否存在。
- 探测 Qdrant 服务。
- 设置 collection 名称和连接参数。
- 调用入库流水线执行文档切片、嵌入生成和向量写入。

`start_store.ps1` 是已有切片的重新同步入口。它跳过切片阶段，直接把现有切片重新写入 Qdrant，通常用于重建 collection、更新向量字段或修复向量库内容。

## 6. 完整报告运行流程

完整报告生成过程可以分为十四个阶段。

### 6.1 环境加载

系统首先读取 `.env`，加载 Qdrant 地址、collection 名称、嵌入模型路径、重排序设置、LLM 设置、IQS 设置、Brain Agent 设置、QA 设置和输出设置。

配置加载遵循“已有进程变量优先”的原则。也就是说，如果外部已经设置了某个环境变量，脚本不会强制覆盖它。这使得项目既能使用本地 `.env`，也能在不同部署环境中通过系统环境变量覆盖配置。

### 6.2 用户问题输入

用户输入可以包含：

- 自然语言研究问题。
- 股票代码或公司标识。
- 是否启用联网搜索。
- 是否启用严格质量模式。
- 搜索预算和输出路径。
- 是否跳过 Reformatter。

在完整报告场景下，系统会把问题传给脑智能体，并将输出模式设置为写作型 Markdown 报告。

### 6.3 问题分析

脑智能体首先执行问题分析。该阶段会识别：

- 研究对象是什么。
- 用户真正要回答的是投资判断、产业分析、公司分析、市场机会、技术路线还是政策影响。
- 问题是否需要最新数据。
- 是否必须联网。
- 是否需要本地知识库参与。
- 报告适合采用什么研究框架。

系统不会固定使用传统的“五大维度”模板，而是根据问题动态生成研究结构。

### 6.4 研究规划

研究规划由 Research Planner 和 Problem Framing 共同完成。它会生成：

- 决策上下文。
- 核心研究问题。
- 待验证假设。
- 反证要求。
- 指标定义。
- 章节候选。
- 证据目标。
- 搜索任务。

如果启用了 LLM 研究规划，系统会尝试让 LLM 生成结构化研究计划；如果规划结果不合格或未启用 LLM，则使用规则化动态种子计划兜底。

### 6.5 预布局

Pre-layout Agent 会把研究计划转成报告蓝图。它重点处理：

- 章节标题和章节问题。
- 每章需要的证据类型。
- 每章最低来源数量。
- 每章是否必须包含 A/B 类权威来源。
- 每章是否需要反证、案例、指标、公司材料或专家观点。
- 表格需求和图文布局意图。

该阶段还会阻止僵化模板化章节。例如，系统会识别并改写“市场规模与增速、竞争格局、政策与监管环境、技术路线与产业链、资本动态”这类固定五段式结构，避免报告看起来像套模板。

### 6.6 搜索任务扩展

系统根据章节和证据目标扩展搜索任务。每个任务都会绑定：

- 所属章节。
- 所属假设。
- 要证明或反驳的问题。
- 目标证据类型。
- 必须包含的关键词。
- 禁止误入的关键词。
- 来源优先级。
- 推荐执行的搜索通道。
- 最低来源等级。
- 是否需要新近信息。

搜索任务不是为了“多搜一点”，而是为了服务具体论证。每个任务都应当对应一个证据缺口或验证目标。

### 6.7 路由决策

脑智能体根据用户设置和问题类型决定路由：

| 路由 | 含义 |
|---|---|
| `local` | 只使用本地 RAG |
| `web` | 只使用联网搜索 |
| `both` | 本地 RAG 和联网搜索同时使用 |
| `all` | 本地 RAG 与多个 IQS 证据通道全量并行 |
| `auto` | 系统根据问题自动判断 |

在完整报告入口中，默认采用偏强的全量路由，也就是本地和联网多通道一起执行。

### 6.8 子智能体并发执行

路由完成后，LangGraph 会把任务分发给多个子节点：

- 本地 RAG 子智能体负责检索本地知识库。
- Web Analysis Agent 负责联网分析。
- 六个 IQS lane 负责不同类型的联网证据。

六个 IQS lane 的职责可以概括为：

| 通道 | 主要来源倾向 |
|---|---|
| 官方数据通道 | 政府、监管、统计、协会、官方发布 |
| 公司材料通道 | 公告、年报、招股书、公司官网、投资者关系 |
| 市场研究通道 | 研报、行业报告、咨询机构、数据库摘要 |
| 新闻事件通道 | 新闻、事件、政策动态、市场变化 |
| 技术产品通道 | 技术路线、产品资料、专利、标准、方案 |
| 客户案例通道 | 招投标、案例、客户、应用场景、落地项目 |

不同通道不是简单重复搜索，而是从来源类型上补齐证据结构。

### 6.9 本地 RAG 检索

本地 RAG 子智能体调用 `search/engine.py`。它会经历：

- 上下文改写：如果问题来自多轮对话，会尝试改写成独立问题。
- 查询规划：生成检索关键词、别名、过滤条件和检索意图。
- 向量库连接：连接 Qdrant，并检查 collection 和向量字段。
- 查询向量化：生成 Qwen dense 向量，必要时生成 BGE-M3 dense 和 sparse 表示。
- 混合召回：同时执行密集向量召回和稀疏词法召回。
- 层级召回：先召回父块，再展开或约束到子块，降低孤立片段误命中。
- 候选合并：使用融合分数把不同召回通道的结果合并。
- 重排序：使用外部 API 或本地 cross-encoder 对候选证据重新排序。
- 证据选择：保留最有支持力、覆盖面和可信度的证据。
- 答案综合：生成有来源支撑的初步回答。
- 答案审查：检查是否存在 unsupported claim、覆盖不足或证据不一致。
- 反思迭代：如果证据不足，改写查询继续检索。
- 追踪记录：保存每一跳检索的输入、候选、证据和判断。

该设计让本地 RAG 不只是“向量相似度 Top K”，而是一个带规划、混合召回、重排、证据选择和迭代反思的检索系统。

### 6.10 联网 IQS 检索

联网搜索由 Web Analysis Agent 和 IQS lanes 执行。它们会根据搜索任务生成查询变体，控制每个通道的搜索预算、来源偏好和结果数量。

联网检索的重点包括：

- 尽量优先找到原始来源。
- 对新闻、政策、市场数据、公司材料进行来源分层。
- 对搜索结果进行去重和重排序。
- 对页面内容进行读取和提取。
- 将网页结果转换成结构化数据点和证据项。
- 标记来源质量、时间、标题、链接、摘要、关键数据和缺口。

联网结果返回后不会直接写进报告，而是先进入统一证据池，等待后续证据治理。

### 6.11 证据合并

所有子智能体完成后，脑智能体进入 merge 阶段。它会把本地 RAG、Web Analysis Agent 和各个 IQS lane 的输出标准化为统一证据格式。

标准化过程会尽量保留：

- 证据文本。
- 来源标题。
- 来源链接或本地文档位置。
- 来源类型。
- 来源等级。
- 所属章节。
- 所属搜索任务。
- 所属假设。
- 证明角色。
- 指标、时间、单位和范围。
- 是否支持、反驳或补充某个判断。

统一证据池是后续写作链路的核心输入。

### 6.12 覆盖率评估与补充检索

系统会对证据池执行 Supervisor Evidence Loop。该循环会检查：

- 每个章节是否有足够证据。
- 核心问题是否有直接证据。
- 是否缺少 A/B 类权威来源。
- 是否缺少反证。
- 是否缺少指标的时间、口径、单位或范围。
- 是否只有新闻而没有原始材料。
- 是否存在任务未执行或通道失败。
- 是否有明显来源单一、论据稀薄或口径冲突。

如果发现缺口，系统会自动生成 follow-up 查询，并再次调用 RAG 或 IQS 通道补充证据。补充检索会受到最大循环次数、重复查询、无新增信号、覆盖率停滞等条件限制，避免无限循环。

### 6.13 结构化分析

证据覆盖达到可用状态后，系统会运行 Analysis Agent。它会从证据包中提炼：

- 关键结论。
- 章节洞察。
- 指标对比。
- 因果链条。
- 冲突点。
- 风险点。
- 可支持和不可支持的判断边界。

该阶段的作用是把“证据集合”转成“可写作的分析结构”，但仍不直接跳过证据约束。

### 6.14 写作、QA 与重排版

写作阶段由 `writer_agent_clean.py` 组织。它先生成结构化写作包，再由最终写作器渲染 Markdown。

初版报告生成后，QA Agent 会检查结构、来源、引用、内部提示语泄露、证据不足、表格质量和结论支撑。必要时 Rewrite Agent 会根据 QA 意见改写。

随后完整报告入口会提取干净证据，调用 Reformatter Agent 对报告进行重排版、引用规范化和公开表达优化。如果 Reformatter 不可用，ReviewAgent 会执行规则化兜底清理。

最终输出通常包含：

- 原始运行状态。
- 写作包。
- 初版 writer Markdown。
- 重排版 clean Markdown。
- 检索 trace。
- 中间证据和 QA 调试信息。

## 7. LangGraph 脑智能体实现

### 7.1 状态模型

脑智能体维护一个全局状态对象。状态中包含：

- 用户问题和路由模式。
- 问题分析结果。
- 研究计划。
- 本地 RAG 子状态。
- 联网分析子状态。
- 六个 IQS lane 的子状态。
- 搜索任务调度表。
- 证据包。
- 结构化分析结果。
- 报告蓝图。
- 章节证据包。
- 证据图谱。
- 微布局计划。
- 表格包。
- 论证单元。
- 决策包。
- 风险包。
- 附录包。
- Writer 报告。
- QA 结果。
- 原始输出。
- 元数据和错误列表。

这种状态模型让每个阶段都能读取上游结构化产物，并把自己的输出写回同一个状态。

### 7.2 图结构

脑智能体的图执行顺序是：

1. 开始。
2. 问题拆解。
3. 路由判断。
4. 根据路由并发执行本地 RAG、联网分析和 IQS lanes。
5. 合并所有输出。
6. 格式化最终响应。
7. 结束。

这种图结构适合复杂研究任务，因为本地资料、联网资料和不同来源类型的联网通道可以并行运行，最后再统一治理证据。

### 7.3 动态路由

动态路由会根据问题中的意图判断是否需要联网。例如：

- 包含最新、新闻、政策、价格、股价、近期、公告等特征的问题通常需要联网。
- 行业空间、财务、增长、竞争、投资判断等问题通常需要本地和联网结合。
- 如果用户明确指定只查本地或只查联网，则遵循用户指定。

自动路由可以减少不必要的联网调用，同时保证需要实时信息的问题不会只依赖本地库。

## 8. 研究计划与搜索任务设计

### 8.1 核心结构

研究计划中最重要的结构包括：

| 结构 | 含义 |
|---|---|
| Hypothesis | 待验证假设，包含证明标准、反证要求、必要证据类型和指标定义 |
| Chapter | 报告章节，包含章节问题、证据需求和最低来源要求 |
| EvidenceGoal | 证据目标，说明要找什么证据来支撑什么问题 |
| SearchTask | 搜索任务，说明由哪个通道搜索、搜什么、为哪个证据目标服务 |
| ResearchPlan | 总体研究计划，整合问题、假设、章节、证据目标和搜索任务 |

### 8.2 假设驱动

系统采用假设驱动的研究方式。它不是先写大纲再找材料，而是先提出要验证的判断，再要求每个判断具备证明标准和反证标准。

一个高质量假设通常需要：

- 明确陈述。
- 可以被证据支持或推翻。
- 有对应指标。
- 有反证触发条件。
- 有来源等级要求。
- 有必要的正反证据组合。

这使得报告更接近研究过程，而不是材料堆砌。

### 8.3 章节驱动

章节不是固定模板，而是围绕研究问题动态生成。每章都需要回答一个明确问题，并绑定不同证据角色。

常见证据角色包括：

- 指标证据。
- 支持性证据。
- 反证。
- 案例证据。
- 来源核验。
- 专家或机构观点。

章节如果没有足够证据，会被标记为需要补充检索，严重时会影响 QA 通过。

### 8.4 搜索任务分派

搜索任务会根据来源需求分配给不同通道。例如：

- 需要政府统计数据的任务分给官方数据通道。
- 需要年报或公告的任务分给公司材料通道。
- 需要行业规模或竞争数据的任务分给市场研究通道。
- 需要近期事件的任务分给新闻事件通道。
- 需要产品和技术路线的任务分给技术产品通道。
- 需要落地案例的任务分给客户案例通道。

这样的分派机制可以提高证据结构的多样性，避免所有任务都落到同一类网页结果上。

## 9. 数据入库技术实现

### 9.1 入库总流程

入库流水线主要由 `ingest_pipeline.py` 编排。流程包括：

1. 定位原始清洗文本目录。
2. 执行文档切片。
3. 生成父子 chunk。
4. 生成嵌入向量和稀疏词法向量。
5. 创建或检查 Qdrant collection。
6. 批量写入向量和 payload。
7. 写入或复用本地缓存。
8. 输出入库统计和预览结果。

该流程既支持从原始文本重新构建，也支持跳过切片、只同步已有切片。

### 9.2 文档清洗

切片模块会对输入文本做多层清洗：

- 移除 HTML 媒体标签和图片残留。
- 移除 Markdown 图片。
- 归一化表格。
- 归一化标题、空白和换行。
- 清除 OCR 占位符和明显乱码。
- 过滤目录、页眉页脚、参考文献、图注、封面信息等低价值内容。
- 对重复行和重复段落去重。

清洗的目标是减少噪声进入向量库，否则检索会召回大量无关片段，影响报告质量。

### 9.3 表格处理

项目对表格不是简单当作普通文本处理，而是会识别：

- 管道表格。
- HTML 表格。
- 类表格行。
- 指标列。
- 年份列。
- 表头和行内容。
- 合并指标单元格。

表格被标准化后，会保留表格家族、表头、行、指标和上下文信息。这对行业研究报告非常重要，因为报告经常依赖规模、产量、价格、利润率、市场份额等结构化指标。

### 9.4 语义切片

切片阶段会先解析文档章节，再按语义块切分。系统会识别：

- 标题路径。
- 段落。
- 表格。
- 列表。
- 指标段。
- 摘要段。
- 结论段。
- 噪声段。

当段落过长时，会按句子和语义边界拆分；当切片数量异常偏少时，会进入更激进的二次切分。相邻切片会保留一定重叠，避免关键上下文断裂。

### 9.5 父子块结构

项目使用父子 chunk 结构：

- 子块用于精确召回。
- 父块用于提供更完整上下文。
- 子块保存前后邻接关系。
- 父块聚合多个相邻子块。

检索时可以先召回父块再定位子块，也可以从子块回溯父块。这样能兼顾召回准确度和上下文完整性。

### 9.6 质量控制

切片模块会为 chunk 计算质量信号，例如：

- 信息密度。
- 噪声比例。
- OCR 噪声。
- 摘要一致性。
- 页码元数据。
- 是否疑似乱码。
- 是否表格粘连。
- 是否过短或过空。

低质量 chunk 会被丢弃或降权，避免进入向量库后污染检索结果。

### 9.7 去重与合并

系统支持精确去重和近似去重。对于重复文本、相似段落、重复表格或 OCR 造成的重复内容，入库阶段会尽量合并或删除。

在清洗文本模式下，一些成本较高的嵌入辅助合并可以关闭，以提升入库速度。

## 10. 向量库与嵌入实现

### 10.1 Qdrant Collection

项目使用 Qdrant 作为本地向量数据库。Collection 中保存：

- 密集向量。
- 稀疏词法向量。
- 可选 BGE-M3 dense 向量。
- 可选 BGE-M3 sparse 向量。
- 文本和检索文本。
- 文档标题、来源文件、章节路径。
- chunk 唯一标识。
- 父 chunk 标识。
- 表格和指标元数据。
- 质量标记。
- 关键词、别名和检索增强字段。

Qdrant collection 会创建必要的向量字段、稀疏向量字段、索引和 payload 索引。如果已存在 collection 但缺少关键字段，系统会提示需要重建或重新同步。

### 10.2 嵌入模型

项目主要使用本地 Qwen embedding 模型生成 dense 向量。另可启用 BGE-M3 作为增强向量模型。

Qwen dense 向量适合语义相似召回。BGE-M3 可以同时提供 dense 和 sparse 表示，用于增强跨表述、跨语言或关键词敏感场景下的检索效果。

### 10.3 稀疏词法向量

除了 dense embedding，系统还构建词法 sparse 向量。它来自：

- 原文关键词。
- 别名词。
- 标题路径。
- 文档元数据。
- 检索增强文本。
- 分词结果。

稀疏向量适合召回专有名词、公司名、产品名、政策名、指标名和数字口径。它可以弥补 dense 向量对精确关键词不敏感的问题。

### 10.4 Payload 设计

每条向量记录不仅保存向量，还保存大量 payload。payload 的作用是：

- 支持过滤。
- 支持证据引用。
- 支持重排序特征。
- 支持上下文回填。
- 支持来源追踪。
- 支持报告附录生成。

这也是系统能在最终报告中追踪证据来源的重要基础。

## 11. 本地 RAG 检索技术实现

### 11.1 查询规划

检索前，系统会构建 Query Plan。Query Plan 会描述：

- 主查询。
- 扩展关键词。
- 别名。
- 行业或公司实体。
- 时间范围。
- 检索意图。
- 是否需要表格或指标。
- 是否需要更高召回数量。

Query Plan 既可以由规则生成，也可以由外部 LLM 增强。

### 11.2 混合召回

本地检索不是单一路径，而是多路召回融合：

- Qwen dense 语义召回。
- 词法 sparse 召回。
- 可选 BGE dense 召回。
- 可选 BGE sparse 召回。
- 父块召回。
- 子块召回。

召回结果会按唯一键合并，并保留每一路召回的分数来源。

### 11.3 层级检索

层级检索的核心思想是：

- 先找可能相关的父块。
- 再从父块关联的子块中找更精确证据。
- 或者通过子块命中后补回父块上下文。

这种方式适合长篇行业资料，因为单个子块可能很准但上下文不足，单个父块上下文完整但粒度太粗。父子结合可以提升可读证据质量。

### 11.4 候选融合

不同召回通道返回的候选会被融合。融合会综合：

- 原始相似度。
- 排名位置。
- 召回通道权重。
- 查询计划匹配度。
- 文档元数据匹配度。
- 标题和正文支持度。
- 关键词命中。
- 任务目标匹配度。

融合后的候选再进入重排序和证据选择。

### 11.5 重排序

系统支持外部 API 重排序和本地重排序。重排序的作用是把“看起来相似”的候选重新排列为“真正能回答问题”的证据。

重排序会考虑问题与候选文本之间的语义相关性，也会结合本地规则分数，例如指标命中、章节匹配、来源质量和表格价值。

### 11.6 证据选择

证据选择阶段会从候选中选出最终用于回答和写作的证据。它关注：

- 是否直接回答问题。
- 是否包含关键指标。
- 是否来源可靠。
- 是否与其他证据重复。
- 是否覆盖不同角度。
- 是否能支持章节论证。
- 是否存在冲突或反证价值。

被选中的证据会转成 EvidenceItem，供后续证据池和写作链路使用。

### 11.7 反思迭代

如果当前检索证据不足，系统会进入 reflection：

- 判断缺少什么信息。
- 改写下一轮查询。
- 调整关键词和检索意图。
- 再次召回和重排。

这种迭代可以提升复杂问题的回答质量，尤其是首轮查询过宽或过窄时。

## 12. 联网搜索技术实现

### 12.1 Web Analysis Agent

Web Analysis Agent 负责处理需要实时信息的问题。它会：

- 扩展联网查询。
- 调用 IQS 搜索。
- 读取网页内容。
- 过滤低质量来源。
- 对结果进行重排序。
- 输出结构化摘要、关键来源和数据点。

它通常用于政策、新闻、市场变化、公司近期事件和事实核验。

### 12.2 IQS 多通道

IQS lanes 的设计目的是把联网搜索拆成不同来源类型。每个 lane 接收被分配的 search tasks，并按自己的来源倾向执行搜索。

多通道设计有三个好处：

- 覆盖不同来源类型，减少单一搜索结果偏差。
- 便于 QA 判断某类来源是否缺失。
- 可以针对不同证据目标设置不同搜索预算。

### 12.3 来源分层

系统会区分来源层级。一般来说：

- 官方、监管、统计、公司公告属于高可信来源。
- 权威机构、行业协会、头部咨询或数据库属于较高可信来源。
- 主流媒体和专业媒体可用于事件与趋势补充。
- 自媒体、转载和缺少出处的内容通常只适合作为线索。

来源分层会影响证据是否能进入核心论证。

### 12.4 联网结果标准化

联网搜索结果最终会被转成统一证据格式。标准化字段包括：

- 标题。
- 链接。
- 来源名称。
- 发布时间或访问时间。
- 摘要。
- 关键事实。
- 关键指标。
- 所属任务。
- 所属章节。
- 来源等级。
- 证据角色。
- 数据缺口。

这使得联网证据可以和本地 RAG 证据一起进入后续证据治理。

## 13. 证据治理机制

### 13.1 统一证据池

证据池是系统中间态的核心。所有来自本地和联网的内容，只有进入证据池并被标准化后，才可能被写作链路使用。

证据池中的每个证据项通常包含：

- 证据文本。
- 来源信息。
- 任务信息。
- 章节信息。
- 假设信息。
- 证明角色。
- 可信度。
- 指标字段。
- 时间口径。
- 引用标识。

### 13.2 证据分级

系统会把证据分成不同用途：

- 核心证据：可以直接支持章节关键结论。
- 支撑证据：用于补充背景、解释机制或辅助说明。
- 线索证据：可信度或直接性不足，不能单独支撑结论。
- 表格证据：适合构建指标表或对比表。
- 附录证据：适合放到来源清单或背景信息中。

低等级或弱证据不能直接进入核心论证。

### 13.3 覆盖率矩阵

Evidence Binder 会生成覆盖率矩阵，检查每章是否满足：

- 最低总来源数。
- 最低权威来源数。
- 是否有反证。
- 是否有指标。
- 是否有案例。
- 是否有不同来源类型。
- 是否有可引用来源。
- 指标是否包含时间、单位、范围和口径。

覆盖率矩阵既用于写作，也用于 QA 和补充检索。

### 13.4 指标归一

行业研究中常见问题是不同来源的指标口径不一致。系统会尝试记录：

- 指标名称。
- 数值。
- 单位。
- 时间。
- 地域范围。
- 统计口径。
- 来源。
- 是否可比较。

如果口径不清，报告中不应把不同来源的数字强行比较。

### 13.5 冲突识别

当不同来源之间出现结论或数据冲突时，系统会记录冲突。冲突不一定是坏事，它可以成为报告的分析点。

冲突处理通常遵循：

- 优先检查来源等级。
- 检查发布时间。
- 检查统计范围和口径。
- 检查是否为转载或二手解释。
- 检查是否来自不同市场、不同样本或不同定义。

如果冲突无法解决，报告应明确说明不确定性，而不是给出过度确定的结论。

## 14. 写作流水线技术实现

### 14.1 Evidence Binder

Evidence Binder 是写作前最关键的模块之一。它负责：

- 收集证据池中的证据。
- 过滤明显不相关证据。
- 将证据绑定到章节。
- 识别核心证据和支撑证据。
- 建立 source registry。
- 建立 source quality map。
- 建立 metric normalization table。
- 建立 coverage matrix。
- 建立 hypothesis evidence map。
- 生成补充检索建议。

它的作用是把“杂乱证据池”转成“可写作证据包”。

### 14.2 Evidence Synthesizer

Evidence Synthesizer 会把章节证据进一步组织成证据图谱。它关注：

- 哪些证据支持同一结论。
- 哪些证据互相补充。
- 哪些证据存在冲突。
- 哪些证据只能作为背景。
- 哪些证据适合进入表格。

证据图谱帮助后续章节论证避免简单罗列来源。

### 14.3 Micro Layout Agent

Micro Layout Agent 负责为章节规划局部结构。它会决定：

- 每章适合什么论述顺序。
- 哪些段落应先给结论。
- 哪些位置需要表格。
- 哪些位置需要反证讨论。
- 哪些证据适合放正文，哪些适合放附录。

它不是做视觉排版，而是做内容结构排版。

### 14.4 Table Agent

Table Agent 负责从证据中构建可公开展示的表格。它会检查：

- 表格目的是否明确。
- 数据是否有来源。
- 指标是否可比。
- 是否存在单位和时间口径。
- 表格结论是否被证据支持。

系统还限制正文表格数量，避免报告堆满表格而降低可读性。

### 14.5 Claim Builder

Claim Builder 会把证据转成论证单元。每个论证单元通常需要：

- 问题。
- 主张。
- 推理。
- 支撑证据。
- 反证或限制条件。
- 可行动含义。

这一步让报告从“资料摘要”变成“有逻辑的研究判断”。

### 14.6 Chapter Argument Agent

Chapter Argument Agent 负责生成每章的论证包。它会把章节问题、证据、论证单元、表格和风险提示整合为可写作结构。

如果某章证据不足，系统可以选择降级表达、加入限制说明，或者在严格模式下触发 QA 阻断。

### 14.7 Decision Synthesis Agent

Decision Synthesis Agent 负责形成面向决策的综合判断。它会整理：

- 关键结论。
- 投资或经营含义。
- 机会判断。
- 约束条件。
- 需要继续跟踪的指标。

该模块强调“结论必须来自证据包”，不能凭空扩展。

### 14.8 Risk Agent

Risk Agent 负责生成风险和不确定性部分。它会基于证据中的反证、冲突、缺口和外部变量，输出风险分类。

常见风险包括：

- 数据口径风险。
- 政策变化风险。
- 竞争格局变化风险。
- 技术替代风险。
- 客户验证不足风险。
- 商业化进度风险。
- 信息来源不足风险。

### 14.9 Final Writer

Final Writer 只消费结构化写作包，不直接自由发挥原始搜索结果。它负责把：

- 报告蓝图。
- 章节包。
- 论证单元。
- 表格包。
- 决策包。
- 风险包。
- 附录包。
- 来源清单。

渲染成公开 Markdown 报告。

它还会过滤内部标签、内部 gap 语言、调试提示和不适合公开展示的词语。

## 15. QA、重写与重排版

### 15.1 QA Agent

QA Agent 会从多个角度检查报告：

- 是否出现内部标签。
- 是否出现“证据不足但仍强行结论”的表达。
- 是否存在无来源关键判断。
- 是否有结构缺失。
- 是否表格质量不足。
- 是否章节深度不够。
- 是否覆盖率不达标。
- 是否有搜索任务或 lane 失败。
- 是否存在公开报告不应出现的调试语言。

QA 会给出评分、错误、警告、重写建议和必要补充检索。

### 15.2 Rewrite Agent

如果 QA 认为需要重写，Rewrite Agent 会根据 QA 意见改写报告。系统不会无条件接受重写结果，而是会比较重写前后的质量分数。如果重写结果更差，就不会替换原稿。

### 15.3 Reformatter Agent

Reformatter Agent 是完整报告入口中的最终整理环节。它会：

- 从 writer package 中提取干净证据。
- 重新组织报告结构。
- 规范引用。
- 清理章节标题。
- 增强公开表达。
- 追加来源附录。
- 校验引用密度和来源质量。

Reformatter 的目标不是重新研究，而是把已有证据和报告内容整理成更像正式文档的最终版本。

### 15.4 ReviewAgent 兜底

如果 Reformatter 调用失败，ReviewAgent 会作为兜底方案执行规则清理。它会移除：

- 内部提示语。
- 空项目符号。
- 重复段落。
- 空章节。
- 截断内容。
- 明显不适合公开展示的残留文本。

## 16. 输出产物

完整报告运行结束后，输出目录通常会产生以下文件：

| 文件类型 | 作用 |
|---|---|
| `state.json` | 脑智能体完整运行状态，适合调试和复盘 |
| `writer_package.json` | 写作流水线结构化输入与中间包 |
| `writer.md` | Final Writer 生成的初版 Markdown |
| `clean.md` | Reformatter 或 ReviewAgent 清理后的最终 Markdown |
| trace 文件 | 本地检索过程、召回候选、证据选择和反思过程 |
| session 文件 | 会话记忆或多轮上下文记录 |

其中 `clean.md` 更适合作为最终交付报告，`writer_package.json` 和 `state.json` 更适合作为技术调试材料。

## 17. 配置体系

项目主要通过 `.env` 控制运行行为。配置可以分为以下几类：

| 类别 | 示例含义 |
|---|---|
| Qdrant 配置 | 服务地址、collection 名称、是否使用 gRPC |
| 嵌入配置 | Qwen embedding 路径、BGE-M3 是否启用、缓存设置 |
| 检索配置 | 召回数量、重排序数量、层级检索、反思轮数 |
| IQS 配置 | 搜索预算、结果数量、页面读取、联网分析开关 |
| LLM 配置 | 规划、分析、写作、QA、重排版等模块是否调用 LLM |
| Brain 配置 | 是否启用深度搜索变体、补充检索循环、最大循环次数 |
| QA 配置 | 严格模式、阻断规则、深度报告评分阈值 |
| 输出配置 | 输出目录、trace 保存、调试快照 |

配置不应在文档中暴露 API Key、Token 或私密服务地址。部署时应通过本地 `.env` 或环境变量注入。

## 18. 异常与降级机制

系统包含多处降级逻辑：

- 如果 LLM 研究规划失败，使用规则化动态计划。
- 如果联网搜索结果不足，进入补充检索循环。
- 如果补充检索没有新信号，停止循环，避免无效消耗。
- 如果 Qdrant 本地路径被锁定，尝试使用服务模式或提示连接问题。
- 如果 Qdrant collection 缺少向量字段，提示重建或重新同步。
- 如果 BGE-M3 不可用，退回 Qwen dense 和词法 sparse。
- 如果外部重排序不可用，使用本地规则重排序。
- 如果 Writer QA 不通过，尝试 Rewrite。
- 如果 Reformatter 失败，使用 ReviewAgent 兜底。

这些降级机制保证系统在部分组件不可用时仍能尽量输出可检查的结果。

## 19. 质量控制重点

项目的质量控制不是只发生在最后，而是贯穿全链路：

| 阶段 | 质量控制 |
|---|---|
| 入库 | 去噪、去重、低质量 chunk 丢弃、表格规范化 |
| 检索 | 混合召回、层级召回、重排序、证据选择 |
| 联网 | 来源分层、页面读取、结果去重、来源过滤 |
| 合并 | 证据标准化、任务绑定、章节绑定 |
| 覆盖 | A/B 来源、反证、指标、案例、来源多样性检查 |
| 写作 | 结构化包约束、禁止凭空生成事实 |
| QA | 内部语泄露、无来源结论、结构深度、表格质量检查 |
| 发布 | Reformatter 和 ReviewAgent 清理 |

最终报告质量依赖每个环节共同保证，而不是仅依赖最终 LLM 写作。

## 21. 推荐运行顺序

首次部署或数据更新时，建议按以下顺序操作：

1. 准备清洗后的文本资料。
2. 确认 `.env` 中的 Qdrant、模型路径和搜索配置。
3. 启动或确认 Qdrant 服务可用。
4. 执行入库流水线，生成 chunk 并同步向量。
5. 检查 Qdrant collection 和向量字段是否完整。
6. 先执行小问题本地 RAG 检索，确认知识库可用。
7. 再执行完整报告链路。
8. 检查输出目录中的 `clean.md`、`writer_package.json` 和 trace。
9. 根据 QA 和覆盖率缺口调整资料、配置或搜索预算。

如果只是重建向量库而不重新切片，可以使用已有切片同步流程。

## 22. 系统边界与维护建议

### 22.1 系统边界

本项目可以显著提升深度报告生成的证据组织能力，但仍有边界：

- 联网搜索质量依赖 IQS 返回结果和可访问网页质量。
- 本地 RAG 质量依赖原始资料清洗质量和入库覆盖度。
- 指标冲突不能总是自动解决。
- LLM 写作仍需要 QA 和引用约束。
- 对实时金融数据、法规和公告，应优先核验原始来源。

### 22.2 维护建议

建议维护时重点关注：

- 定期检查 Qdrant collection 字段是否与代码期望一致。
- 更新模型路径后重新验证嵌入维度和 collection 配置。
- 对新增资料先检查清洗质量，再入库。
- 对重点行业建立 golden cases，防止检索质量回退。
- 对最终报告保留 `state.json` 和 `writer_package.json`，便于复盘。
- 严格保护 `.env` 中的密钥和私有服务地址。
- 当报告反复出现证据不足时，优先补资料或提高搜索预算，而不是只调写作提示词。

## 23. 一句话总结

该项目的核心不是“让模型直接写报告”，而是先用本地 RAG 和联网 IQS 构建可追溯证据，再通过 LangGraph 多智能体把问题拆解、检索、证据治理、结构化写作、QA 和重排版串成一条受控流水线。最终报告的可信度来自证据池、覆盖率检查、结构化写作包和发布前质量门禁的共同约束。
